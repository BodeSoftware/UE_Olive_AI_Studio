Use these Blueprint creation/editing reliability rules:

1. Read before write:
- Use blueprint.read (or specific read tools) before creating or modifying variables, functions, graphs, dispatchers, and nodes.
- Confirm the target Blueprint type and parent class first.

2. Variable typing:
- Use concrete UE types that resolve cleanly.
- For object/class/interface/struct/enum variables, ensure the referenced type exists before creating.
- For containers (array/set/map), ensure nested element/key/value types are explicit and valid.
- Never use wildcard/exec as variable types.

3. Delegates and dispatchers:
- For multicast events, use blueprint.add_function with function_type="event_dispatcher" (do not model dispatchers as raw variables).
- Avoid creating single-cast delegate member variables unless explicit tool support exists.

4. Graph edits:
- Prefer surgical edits (add/connect/remove specific nodes) over rebuilding full graphs.
- After structural edits, run compile and inspect warnings/errors.
- If compile warnings mention missing signatures, invalid pin types, or unresolved classes, fix type/modeling first before further edits.

5. Error recovery order:
- First retry by re-reading current asset state.
- Then adjust only the failing operation.
- If the same signature repeats, stop and propose a safer alternative plan to the user.

6. Naming and safety:
- Follow project naming conventions and existing patterns in the target asset.
- Avoid destructive changes unless explicitly requested.

7. No fake success:
- If batch_write returns an error, report what failed truthfully
- Do not summarize partial failures as successes
- If you cannot verify, say so explicitly

## Blueprint Creation Quick Reference

Recommended workflow order:
1. CREATE the Blueprint (path must end with asset name: /Game/Blueprints/BP_Name)
2. ADD components (StaticMeshComponent, CapsuleComponent, etc.)
3. ADD variables (types: Float, Boolean, Vector, Int, String, etc.)
4. BUILD graph logic with blueprint.apply_plan_json (schema_version "2.0") — one call for all logic
5. VERIFY with blueprint.read

PATH FORMAT:
  ✓ /Game/Blueprints/BP_Gun
  ✓ /Game/Characters/BP_Enemy
  ✗ /Game/Blueprints          ← FOLDER, not asset
  ✗ /Game/Blueprints/         ← Trailing slash = folder
  ✗ Blueprints/BP_Gun         ← Must start with /Game/

VARIABLE TYPES (use these exact strings):
  Float, Boolean, Int, String, Vector, Rotator, Transform,
  "TSubclassOf<Actor>", "TArray<Float>", Name, Text, Color

PLAN JSON OPS:
  event, custom_event, call, get_var, set_var, branch, sequence,
  for_loop, for_each_loop, while_loop, do_once, flip_flop, gate,
  delay, spawn_actor, print_string, is_valid, cast, make_struct,
  break_struct, return, comment

## Multi-Asset Task Planning

When the user's request requires creating MULTIPLE Blueprints (e.g., "create a gun and bullet system"):

1. Before calling any tools, list all assets you need to create
2. Completing one asset at a time (structure + graph logic + compile) before starting the next tends to work better
3. Allocate your iteration budget: if you have 10 iterations and need 2 assets, spend at most 5 on each
4. Avoid spending all iterations on one asset when multiple are needed
