## Niagara System Patterns

### Hierarchy
- System > Emitter > Stage > Module
- A system contains one or more emitters, each with its own stage stacks
- System-level stages (SystemSpawn, SystemUpdate) run once for the whole system
- Emitter-level stages run per emitter; particle-level stages run per particle

### The 6 Stages
- SystemSpawn — runs once when system activates (system-wide init)
- SystemUpdate — runs every frame at system level (system age, global parameters)
- EmitterSpawn — runs once when emitter activates (emitter init)
- EmitterUpdate — runs every frame per emitter (spawn rate, emitter lifetime)
- ParticleSpawn — runs once per new particle (initial size, color, velocity, lifetime)
- ParticleUpdate — runs every frame per living particle (forces, aging, fading)

### Common Modules by Stage
- **EmitterUpdate**: Spawn Rate, Emitter State, Spawn Burst Instantaneous
- **ParticleSpawn**: Initialize Particle (should be first), Add Velocity, Shape Location
- **ParticleUpdate**: Gravity Force, Drag, Curl Noise Force, Scale Color, Scale Sprite Size, Update Age, Solve Forces and Velocity

### Module Ordering
- Initialize Particle first in ParticleSpawn — sets baseline attributes other modules depend on
- Force modules (Gravity, Drag, Curl Noise) before Solve Forces and Velocity
- Solve Forces and Velocity integrates accumulated forces into position — must be after all force modules
- Update Age should run each frame in ParticleUpdate for lifetime-based curves to work
- Scale Color / Scale Sprite Size use normalized age for over-lifetime effects

### Renderers
- Sprite Renderer — camera-facing quads (most common, fire/smoke/sparks)
- Mesh Renderer — instanced static meshes (debris, projectiles)
- Ribbon Renderer — connected trail strips (sword trails, lightning)
- Light Renderer — dynamic point lights per particle (use sparingly)

### Available Tools
- niagara.create_system — create a new empty system asset
- niagara.read_system — inspect current structure (emitters, stages, modules, renderers)
- niagara.add_emitter — add an emitter (empty or from a source emitter template)
- niagara.add_module — place a module in a specific stage at a given stack position
- niagara.remove_module — remove a module by its ID
- niagara.list_modules — search available modules by name or filter by stage
- niagara.compile — compile the system (async, returns immediately)
- niagara.set_parameter — Set a module parameter value (float, int, vector, color). RIP types update instantly without recompile.
- niagara.describe_module — List all parameters for a module (names, types, defaults, current overrides).
- niagara.set_emitter_property — Set emitter-level properties (SimTarget, bounds mode, determinism, etc.). Requires recompile.

### Workflow
- Read the system structure before modifying an existing system
- Use niagara.list_modules to discover module names when unsure of exact spelling
- Add modules in dependency order (Initialize Particle before velocity/force modules)
- Compile after making changes — compilation is async and does not block

### Parameter Setting
- Use `niagara.describe_module` first to see available parameters and their types
- Numeric parameters (float, int, vector, color) use Rapid Iteration — no recompile needed
- Bool and enum parameters are not yet supported via set_parameter
- Vector values: comma-separated "X,Y,Z" (e.g., "0,0,100" for upward velocity)
- Color values: comma-separated "R,G,B,A" (e.g., "1,0.5,0,1" for orange)
- Always use float-precision vectors (FVector3f, not FVector)

### Emitter Properties
- Common properties: SimTarget (CPUSim/GPUComputeSim), CalculateBoundsMode, bDeterminism, RandomSeed
- Set via `niagara.set_emitter_property` — requires recompile after changes
- GPU simulation: set SimTarget to GPUComputeSim for high particle counts (10k+)
