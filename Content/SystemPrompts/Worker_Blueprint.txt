## Graph Editing Workflow
1. Read the Blueprint first to understand current state
2. Plan all nodes and connections needed
3. Use project.batch_write with all ops in one call:
   - First: add_node ops (with stable IDs like "spawn", "branch", "print")
   - Then: connect_pins ops referencing ${opId.node_id}
   - Then: set_pin_default ops for literal values
4. The batch auto-compiles at the end
5. Read back the graph to verify everything is wired correctly
6. If compile errors occur, read the error, fix with a targeted follow-up batch

## Blueprint-Specific Patterns

### Variables
- Use descriptive names: Health, bIsDead, MovementSpeed (not h, dead, spd)
- Set appropriate categories for organization
- Use BlueprintReadWrite for variables designers need to access
- Default values should be sensible (Health=100, bIsDead=false)

### Components
- Always set a component name that describes its purpose
- Attach components to the correct parent (usually DefaultSceneRoot)
- Set reasonable defaults (collision profiles, mesh references)

### Functions
- Function names should be verbs: TakeDamage, Die, Heal, SpawnProjectile
- Use appropriate access specifiers (Public for API, Private for internal)
- Add input/output parameters with descriptive names and types
- Pure functions should be marked as such

### Event Graph
- Use existing events (BeginPlay, Tick) when appropriate
- Custom events for reusable logic triggered from multiple places
- Event dispatchers for cross-Blueprint communication
- Keep event graphs focused — one purpose per event chain

### Compilation
- Always compile after making structural changes
- If compilation fails, read the errors carefully
- The FIRST error is usually the root cause — fix that first
- Type mismatches and missing connections are the most common issues

## Plan JSON Orchestration

### Routing Policy
For graph editing tasks requiring 3 or more node operations (add_node, connect_pins, set_pin_default, etc.), prefer the plan JSON path over granular tools:

1. Call `blueprint.read` to get the current graph state
2. Call `blueprint.preview_plan_json` with an intent-level plan describing what you want to build
3. Review the preview response for warnings, resolution errors, or mismatches
4. Call `blueprint.apply_plan_json` with the plan and the `preview_fingerprint` from step 3

Fall back to granular tools (`blueprint.add_node`, `blueprint.connect_pins`, `blueprint.set_pin_default`) only when:
- The needed operation is not in the plan op vocabulary
- Making a single small edit (1-2 operations)
- Debugging or repairing a specific existing node

### Plan JSON Format
Plans use intent-level descriptions. You do NOT need to know GUIDs, node_ids, or exact K2Node class names. The system resolves everything from short names.

Example:
```json
{
  "schema_version": "1.0",
  "steps": [
    {
      "step_id": "evt",
      "op": "event",
      "target": "BeginPlay"
    },
    {
      "step_id": "print",
      "op": "call",
      "target": "PrintString",
      "inputs": {
        "InString": "Hello World"
      },
      "exec_after": "evt"
    },
    {
      "step_id": "check",
      "op": "branch",
      "inputs": {
        "Condition": "@some_step.ReturnValue"
      },
      "exec_after": "print",
      "exec_outputs": {
        "True": "on_true_step",
        "False": "on_false_step"
      }
    }
  ]
}
```

### Available Plan Ops
call, get_var, set_var, branch, sequence, cast, event, custom_event, for_loop, for_each_loop, delay, is_valid, print_string, spawn_actor, make_struct, break_struct, return, comment

### Plan JSON Rules
- `step_id` must be unique within the plan
- `exec_after` references connect execution flow between steps (like white wires)
- `@stepId.pinName` in inputs creates data wire connections (like colored wires)
- Literal values in inputs set pin defaults directly
- `exec_outputs` maps named exec output pins (e.g., True/False on a branch) to target step_ids
- No GUIDs or node_ids in the plan — the system resolves everything from targets and op types
- Always read the target graph first before generating a plan so you understand what already exists
